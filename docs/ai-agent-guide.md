## 1. Agent 的定位

Agent 是**唯一具备语义理解能力的审校者**。对齐器和自动修复器都基于规则，不理解文本含义。

| 层级        | 执行者                      | 职责                        |
| ----------- | --------------------------- | --------------------------- |
| L1 对齐     | `align()`                   | 嵌入向量 + DP 行级对齐      |
| L2 自动修复 | `RepairService.auto_repair` | 按策略矩阵确定性修复        |
| L3 AI 审核  | **AiRepairAgent**           | 语义判断 L2 结果是否正确    |
| L4 人工校验 | GUI 用户                    | 最终决策，可撤销 agent 判断 |

**核心原则**：规则算法用最低成本解决最简单的问题；留给 agent 的是需要判断力的工作。

## 2. 两层文本模型

```
初始文本 (对齐器原始输出)
  ── 自动修复注入 ──→ 当前文本 (AI 审校对象)
```

| 层           | 来源                                  | AI 的作用                            |
| ------------ | ------------------------------------- | ------------------------------------ |
| **初始文本** | `AlignmentSnapshot.original_ops`      | 操作基准——edit/merge/delete 都基于此 |
| **当前文本** | 初始文本 + 自动修复 = `replay()` 结果 | 审校对象——AI 判断"这对吗？"          |

**关键约束**：

- AI 不需要知道当前文本的来源（自动修复、手动、预览）
- AI 不需要理解 auto*note、would*\*、策略名等机器内部概念
- 所有工具操作的是初始文本，系统负责"操作初始文本 → 重新 auto-repair → 更新当前文本"
- `ok` 只表达"当前文本正确"，系统自行决定如何归一化为 1:1

## 3. AI 工具接口

### 3.1 工具列表

| 工具     | 参数                                         | 语义                               |
| -------- | -------------------------------------------- | ---------------------------------- |
| `ok`     | `snap_id=N`                                  | 当前文本正确，按当前文本输出 1:1   |
| `edit`   | `snap_range=N, new_src=[...], new_tgt=[...]` | 替换初始文本的指定侧内容           |
| `merge`  | `snap_range=N` 或 `"i-j"`                    | 将选中 snap 的初始文本拼接为一段   |
| `delete` | `snap_range=N` 或 `"i-j"`                    | 清空选中 snap 的全部内容           |
| `flag`   | `snap_id=N, note`                            | 标记待人工处理                     |
| `view`   | `pair_spec="1-3,5,11"`                       | 查看文本对的完整内容               |
| `append` | `snap_id=N`                                  | 将新发现的异常 snap 追加到待审列表 |
| `done`   | `note=""`                                    | 结束审校对话                       |

### 3.2 使用规则

- **没有 `split` 工具**——拆分应当用 `edit` 手动实现
- **`edit` 只传需要修改的一侧**
- **`merge` 和 `delete` 支持连续范围 `"i-j"`**
- **`edit` 替换的是初始文本**，不是当前文本

### 3.3 AI 决策树

```
当前文本的每一行 src/tgt 语义对应、内容正确吗？
  ├── 是 → ok（系统自动归一化为 1:1）
  ├── 多出不需要的行 → delete
  ├── 应合并为一段 → merge
  ├── 文本需要修正 → edit
  └── 不确定 → flag

对于非 1:1 snap：检查相邻行是否存在错位
  ├── 有 → append 追加到待审列表
  └── 无 → 按上述决策树处理
```

## 4. 消息构造

异常 snap 展示初始+当前两层，`>>` 标记异常 snap，上下文行以空格开头：

```
>> [3] 初始: 1:0 → 当前: 1:1
   初始 src:
    #0  扫图：撸管娘
   初始 tgt:
    #0  (空)
   当前 src:
    #0  扫图：撸管娘
   当前 tgt:
    #0  Image Scanning: Lulu Niang

   [24] 1:1
    src: （干吗待在那边淋雨？）
    tgt: (Why is she sitting there getting soaked?)
```

AI 通过 `view()` 按需拉取详情，初始消息只含统计概览和异常对 ID 列表。

## 5. merge 与 split 的决策框架

### 两类 N:1 场景

| 场景             | 特征                                         | 正确操作       |
| ---------------- | -------------------------------------------- | -------------- |
| **原文意外断行** | 原文两行显然应连为一句，译文一行是正常翻译   | **merge 原文** |
| **翻译语义融合** | 原文两行语义独立，译文是译者将两者融合为一句 | **split 译文** |

> 原文优先的前提下，原文本身的断行错误应通过合并原文而非拆分译文来修正。

### 策略绑定决策

- `src-first`：优先保持原文不动，调整译文侧。原文意外断行时先 edit/merge 修正原文。
- `tgt-first`：优先保持译文不动，调整原文侧。
- `minimal`：优先用合并替代拆分，不引入新信息。

## 6. 典型情景与正确行为

### 情景 1：自动修复完全正确

N:1 → 拆分译文后子句语义完整 → `ok`

### 情景 2：自动修复方向错误

原文断行，auto_repair 拆分了译文 → 跨 snap merge 原文

### 情景 3：跨 snap 译文错位

snap[25] 译文是插图标记，snap[26] 译文是整体翻译 → 整体性错误，delete 或跨 snap merge

### 情景 4：缺失译文

N:0，含 `⟢MISSING⟣` 占位符 → `edit` 补译（绝不能用 `ok`）

### 情景 5：翻译质量问题

译文有杂糅或用词不当 → `edit(new_tgt=[...])`，保留原文

### 情景 6：低分但内容正确

score=35% 但实际对应正确 → `ok`

### 情景 7：无法处理

原文缺失、专有名词无可靠翻译 → `flag(note="原因")`

## 7. 提示词设计原则

### 必须告知 Agent 的

1. 对齐器和自动修复器的背景—理解数据来源
2. `⟢MISSING⟣` = 绝不能 ok
3. 上下文是辅助判断工具，不是待办项
4. 评分的含义和局限性

### 必须避免的

1. 暗示"大多数是正确的"—导致过度保守
2. 暴露算法实现细节（DP、互惠匹配等）
3. 暗示"应该信任程序"—agent 是最后防线

### 不应做的

1. 不改变原始文本的风格
2. 不批量删除明显正确的文本对
3. 不为节省轮次而省略必要的查看
