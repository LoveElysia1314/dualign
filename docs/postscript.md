# 后记

这个项目最初和"文本对齐"没有任何关系——只是我在备考六级，同时在读一部轻小说。某天冒出一个念头：如果一行中文、一行英文对照着排，是不是顺便就能把英语学了？我搭了一条简单的管线：爬取章节，调翻译 API，拼成双语对照页面。然后问题就来了——翻译 API 返回的行数和原文永远对不上。两行中文缩成一行英文，一行英文又拆成两行中文，偶尔还会跳过一整行。我一头扎进了这个"对齐"问题，投入了大量精力，以至于在六级考场上铩羽而归。

但这颗种子没死。后来我租了一台云服务器，注册了域名，部署了个人主页，又把这个项目捡了起来。原本打算做成静态网页的双语阅读器，放在另一个项目（双语轻小说站）上，却反过来给了我把对齐器继续做下去的动力。经过数次重构，这个模块最终独立了出来，有了现在的名字——Dualign (Dual-align)，使命很简单：让平行文本对齐变得简单、可靠、可扩展。

附带一提，项目徽标是 ChatGPT 给的灵感，我使用 Adobe Illustrator，在端午节期间绘制的。设计经过几轮迭代后定型的。图标左侧是一个 3×2 矩阵，右侧是五个六边形——分别象征相似度矩阵和平行文本对的对齐关系（2:1 + 1:1）。我有点强迫症，画布定在 1080×1080，所有元素的边长或多边形外接圆半径我都喜欢用 2 的幂次或 24 的倍数，它们天然有一种秩序感。最终的结果，我尽量做到了像素客观居中、视觉重心也协调，还是有一点喜悦的。

## 与 AI 同行

这一个半月的高强度开发，从头到尾是纯 vibe coding。AI 是我最重要的工具。它帮我把自然语言变成代码，帮我分析时间复杂度，帮我设计测试用例，写文档，写博客——这些是效率层面的事。更关键的是，它给了我很多我不知道的东西。比如它告诉我 sentence-transformer 是平行文本对齐的重要方案，我第一次体悟到 embedding 的思想有多优雅；它替我读了模型官方文档，指出 Instruct 前缀对 qwen3-embedding 的质量有显著影响——这些我靠自己是不会去翻的。

但话说回来，偶尔也让我感到欣慰的是，我偶尔也能在 BUG 修复上发挥作用。筛选锚点的思路、填充分段的规则、状态管理的设计——这几件事是我自己提出的。一个我印象很深的例子是，旧架构里有个索引漂移问题，AI 在几十次对话里反复修，始终跳不出局部。最后是我自己跳出来，设计了不可变快照加追加日志的方案：原始对齐结果永远封存，所有修改只追加不修改。这套设计让我在 AI 的局部视野之外，重新找回了一点全局掌控感。

说来惭愧，我完全不理解现在代码的具体架构；但不知是否在看 AI 修 BUG 的过程久病成医，我有时仅凭程序的表观逻辑，就能猜到代码的逻辑错误所在。

当然，和 AI 协作不是只有高光时刻。项目全部代码和大多文档都是 AI 生成，理解偏差会不断积累。我没有审查代码的能力，于是一个 bug 反复出现，修了又冒，修了一个又冒出一个。于是我不得不一遍遍组织核心思想跟 AI 沟通，或者不断测试各种边缘场景，发现新问题再解决。但恰好是这个循环，让我在完全没读过源码的情况下，对项目的算法和实现有了相当深入的理解。

AI 帮你落地代码，但算法的核心思想是你自己的。工具的形态是演化出来的，不是设计出来的。只是在每一个分岔路口，选择哪条路这件事，还是开发者自己的活。

## 写在最后

工具方面，这个项目主要靠 DeepSeek API，接入 VS Code 的 Copilot（顺便，VS Code 近期已经支持用户直连第三方 API，不用靠第三方插件了）。算上另一个不公开的项目——双语轻小说站的翻译校订批量管线——还有一些个人零散使用，据保守统计，DeepSeek API 调用超过了 7 万次，token 消耗超过了 120 亿，成本超过 800 元。当然，VS Code 的 agent 架构（以及调用插件接入第三方api造成的经常性请求错误）使得上下文和调用次数有不小水分；再加上 DeepSeek v4 接近免费的缓存命中收费和约 99% 的缓存命中率，这个夸张的 token 数倒不至于对应天文数字的账单。

开销大得吓人，而且这个项目也不太可能产生什么经济效益。但我还是觉得，在 AI 时代尽早熟悉和适应这些工具——比如用 API 而不是网页端对话——是一笔划算的投资。用 AI 过程中拓宽的知识面和培养的思维方式，对个人能力的提升是实打实的。

需求导向的开发，尤其当需求来自个人兴趣和实际问题，往往能激发更大的热情。我开源仓库里大部分都是这样的项目：某个游戏非官方服务器的登录器，一个小众软件的插件，一个个人博客的主题……没多大商业价值，但它们满足了我自己的兴趣，这就足够给我动力去做好。

这样一个起源于一时兴起的思维火花，落地成了一个精致的"玩具"。算法本身不复杂，在GUI和交互逻辑上也走了不少弯路，但我依旧有种别样的满足感。

## 附录 A：旧实现——精巧的弯路

在 Dualign 之前，曾有一个叫 [Bilingual Aligner](https://github.com/LoveElysia1314/bilingual_aligner) 的前身。那时我试图用手工规则掌控一切。

我把源文与译文铺成网格，用长度比例裁剪，用标点差异惩罚，用阈值剪枝。为了让动态规划跑得快一点，又套上层层约束，五个环节串成一串。参数之间相互耦合，改一个就牵动全局，出了错很难定位到根源。后来我才意识到，两百行乘两百行的全量余弦相似度矩阵，NumPy 连一毫秒都用不到——真正的瓶颈是嵌入 API 调用，每个文本段编码都需要将近一百毫秒。我花大力气优化的，是一个并不存在的瓶颈。

还有一个标点惩罚。我写了一个二次衰减函数，用中英文标点数量的差异来调节权重。逻辑上似乎说得通，但让平行文本得分更高这件事，早已被 sentence-transformer 内化在嵌入向量里了。手工特征在抢模型的活。

合并评分也有问题。多行合并成一行时，旧版把各行的嵌入向量取平均再算余弦相似度。"门开了。她走进来。"取平均后，"开门→走进"的因果链就没了——这两句话在向量空间里不是两个点的中点，而是一个新的方向。

但更根本的局限藏在搜索空间里。旧版把所有候选边丢进全量 DP 公平竞争，没有任何先验信息。这迫使它把移动方式死死限制在 1:1、2:1、1:2 三种。想支持 N:1 或 1:N？合并候选的编码成本随文档长度平方级增长，根本不可能在全量范围内展开。设计精巧，但灵活性被自己锁死了。

这些弯路的根源是同一个习惯：手工结构太多，抢了模型该干的活，也框死了可能性。精巧，但并不实用。

## 附录 B：新实现——锚点与分治

Dualign 做的核心改变，是一次方向性的翻转：不再在全量空间中暴力搜索，而是先找到一批可信的参照系——锚点。

锚点的筛选基于信任余量的双边互惠：一对句子要想成为锚点，必须有足够的相似度余量，且两侧互相认为对方得分在可接受范围内。margin 从 0.01（低分严苛）到 0.05（高分宽容）随 top-1 得分线性变化，天然适配不同的文本相似度分布。效果出奇地好：平行文本中锚点密度接近 90%；非平行文档中锚点稀疏，进而被对齐器拒绝。

更重要的是，锚点将长序列切成一串独立段，问题从"全局 DP"变成了"锚点之间的段怎么填"。核心规则：以每个锚点为基准，向自由行方向扩展——连续、至少两行、恰好包含一个基准行、不超过长度上限 θ（默认 20）。N:1 和 1:M 在这个过程中自然涌现，无需单独设计搜索空间。残余区域通过受限 DP 补全赝锚点，整个流程分阶段收敛。最终只运行一次 DP，不需要硬编码合并上限。

合并评分也不再取平均，而是把多行拼接后重新编码——让模型读完整的段落，而非审视割裂的切片。语义的顺序本身就是信息，不该被平均数压缩掉。面对无法匹配的孤行，Dualign 选择了不惩罚：承认缺失，远比制造虚假的 1:1 安全。

系统在功能膨胀后分成了四层——算法锚定、规则缝合、AI 审视、人类决断——每层恪守边界，各司其职。

> 关于状态管理的补充：旧架构中 AI 反复修不好的索引漂移问题，在正文已有叙述。这里补一句技术细节——最终方案是不可变快照（`AlignmentSnapshot`）加追加日志（append-only `RepairAction[]`），`undo()` 只需移除日志最后一条，`replay()` 是纯函数。没有任何逆操作。
